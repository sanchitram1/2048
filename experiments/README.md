# Experiments playbook

This round is about finding the strongest offline imitation checkpoint. Based on previous
experiments, the RL agent is only good at scaling if the basin is right, and we're not
at the right basin yet.

## What has changed since then?

We have clean schema-v2 teacher-label data uploaded to 
[Google Drive](https://drive.google.com/drive/folders/1NUsKYElcfc9oQAtUQtsQbmim_kTd5jLk?usp=drive_link) 
and shared:

- Expert-iteration archive:
  `expert-latest-2.3m-stages2-scenarios10-2026-05-07.tar.gz`
  - MCTS labeled boards, based on examples that the current_best_checkpoint actually 
    encountered
  - 2.3M rows
  - 46 shards
  - MCTS relabeling with `stages=2`, `scenarios=10`
- MCTS self-play archive:
  `mcts-dataset-latest-725k-stages2-scenarios10-2026-05-07.tar.gz`
  - 724,992 rows
  - 177 shards
  - generated by raw MCTS self-play
  - MCTS labels with `stages=2`, `scenarios=10`

The problem was bad / inconsistent labels that has been fixed. The key question now is 
**Which clean data distribution gives the best multihead imitation checkpoint?**

## Rules for this round

Keep all of these fixedL

- Architecture: `--value-network qnetwork`
- Total train rows for comparable runs
- Same train/val split policy and split seed
- Same epochs budget
- Early stopping enabled
- Same learning rate, batch size, LR schedule, soft-target weight, and patience
- Same diagnose command, episodes, and eval seed



4. Prepare mixture

Some experiments are going to blend the data. I found that the expert-iteration data is
good at middle games, since it's cases that the current agent encounters. The MCTS 
self-play archive is really good at end games, since its cases that our agent has never
seen before.

```bash
merge-mcts-datasets blend \
  --rows 500000 \
  --output data/mix_500k_expert400k_mcts100k_seed1000.npz \
  --seed 1000 \
  data/expert-latest:80 \
  data/mcts_dataset_latest:20
```

5. Fixed imitation command

Imitate helps us generate the MCTS copy. Change only `--labels`, `--save-step`, and 
output directory. 

```bash
imitate --train-only \
  --labels data/mix_500k_expert400k_mcts100k_seed1000.npz \
  --value-network qnetwork \
  --epochs 50 \
  --batch-size 128 \
  --learning-rate 0.0001 \
  --soft-target-weight 0.5 \
  --val-fraction 0.1 \
  --split-seed 1000 \
  --split-by-source \
  --log-agreement-every-epoch \
  --early-stop-patience 5 \
  --early-stop-min-delta 0.0 \
  --lr-schedule cosine \
  --warmup-epochs 1 \
  --imitation-run-dir models/scratch/<run_id> \
  --model-dir models/scratch/<run_id> \
  --save-step <run_id> \
  --device auto
```

Use `models/scratch/<run_id>/checkpoint_best.pt` for evaluation.
`checkpoint_final.pt` is the last trained epoch, not necessarily the best
checkpoint.

6. Evaluation command

Run the same diagnose for every candidate:

```bash
diagnose \
  --checkpoint models/scratch/<run_id>/checkpoint_best.pt \
  --eval-base-seed 1000 \
  --episodes 250
```

In PR, report:

- run id
- exact labels input
- total rows used
- data mixture
- `checkpoint_best.pt` path
- best epoch and best `val_teacher_exact`
- diagnose mean, median, max
- counts for 256, 512, 1024, 2048

## Experiments

| ID | Moving part | Labels input | Target rows | Goal |
|----|-------------|--------------|-------------|------|
| I0 | 100/0 expert/MCTS | prepared 500K expert subset | 500K | Measure clean expert-distribution ceiling at equal row count | 
| I1 | 95/5 expert/MCTS | prepared mixture | 500K | Check whether a small amount of raw MCTS states helps | 
| I1 | 90/10 expert/MCTS | prepared mixture | 500K | Same total rows, more MCTS | 
| I2 | 80/20 expert/MCTS | prepared mixture | 500K | Current main hypothesis | 
| I3 | 60/40 expert/MCTS | prepared mixture | 500K | Stress-test whether MCTS distribution dominates | 
| I4 | full expert-only scale | `data/expert-latest` | 2.3M | See whether scale alone improves middle-game play |
| I5 | full MCTS-only | `data/mcts_dataset_latest` | 200K | See whether scale on MCTS alone improves end-game play |

We won't compare I4 and I5 with everything else, since the difference in scale is huge, 
but it should give us inputs directionally. 

## Separate Experiment

Side note, if anyone has some server / GPU they can just run this script on, or even on 
their laptop in the background, that would be helpful. This is separate from training.

```bash
mcts-dataset \
  --output-dir data/mcts_dataset_<your_name>_<seed> \
  --games 0 \
  --stages 2 \
  --scenarios 10 \
  --shard-rows 4096 \
  --seed <unique_seed>
```

Leave it running. The end goal is at least 100K rows per generator; 1M rows is
better. Different seeds are useful because they create different game
trajectories and give us more board coverage.

Once you run the above, try this: 

```bash
python - <<'PY'
import json
from pathlib import Path
import numpy as np

run = Path("data/mcts_dataset_<your_name>_<seed>")
m = json.loads((run / "manifest.json").read_text())
rows = 0
bad_schema = []
for name in m["shard_files"]:
    z = np.load(run / name, allow_pickle=False)
    if int(np.asarray(z["schema_version"]).item()) != 2:
        bad_schema.append(name)
    rows += int(z["boards"].shape[0])
print("rows", rows)
print("shards", len(m["shard_files"]))
print("bad_schema", bad_schema)
print("keys", sorted(np.load(run / m["shard_files"][0], allow_pickle=False).files))
PY
```

Make sure the keys include `teacher_q`, `teacher_policy`, `board_hash`,
`source_indexes`, `episode_id`, and `move_idx`. If they don't include this, kill the 
process and message on WhatsApp. 

## TLDR

- These are `imitate` experiments
- No RL experiments.
- Do not change architecture.
- Do not tune learning rate, batch size, or patience while claiming a data result.
- Do not use old action-only labels.
- Do not report a run without fixed-seed diagnose.
